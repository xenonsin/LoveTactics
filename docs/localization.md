# Localization — design & architecture

**Status:** implemented. The catalog is a single *columns-per-language grid*
(`data/lang/strings.lua`) behind the `Locale.get` seam. This doc records the design and the
rationale for choosing the grid over per-language files (that history is kept below because the
trade-off may need revisiting at scale).

## Context

Story is delivered visual-novel style (`ui/dialogue.lua`, `models/conversation.lua`). Text is
authored **inline in English** in the conversation files; a translator's catalog supplies other
languages. Two design pillars, both already in place and unchanged by this plan:

- **Authoring stays inline & readable.** A conversation is a script of `{ "speaker", "text" }`
  nodes. Authors never type keys.
- **Stable ids, generated by a tool.** `& "E:\LOVE\lovec.exe" . extract-strings`
  (`tools/extract_strings.lua`) stamps a stable `tag` into each line and generates the translation
  catalog. The key schema lives in one place — `Locale.key` in `models/locale.lua`
  (`line.<conv>.<tag>`, `title.<conv>`, `name.<id>`) — so the tool and runtime always agree.

### The catalog: one columns-per-language grid

`data/lang/strings.lua` — **one row per string id, a column per language** (the "localization
spreadsheet" layout, `id | en | ja | …`):

```lua
-- data/lang/strings.lua   (generated + merged by extract-strings; `en` is a mirror column)
return {
  ["line.wrath_intro.1"] = { en = "So. Fresh blood for the sand...", ja = "ほう、砂場に..." },
  ["name.colosseum"]     = { en = "The Colosseum",                    ja = "闘技場" },
}
```

Runtime resolves via `Locale.get(key, fallback)`: in English it returns the inline fallback;
otherwise the active language's cell, else (missing/blank cell) the inline English.

> This replaced an earlier *per-language-file* layout (`en.lua` + `ja.lua`); the trade-off write-up
> below explains why, and `extract-strings` migrated the old files into the grid in one run.

## Why (and why it isn't simply "better")

The real distinction is **edit-format vs runtime-format**, not one layout beating the other:

- A **columns grid** is the better *edit/source* format — the translator sees English and their
  language in the same row, and missing cells make coverage obvious at a glance.
- **Per-language files** are the better *runtime* format — load only the active language; if humans
  hand-edit, two translators never touch the same file.

The professional pipeline uses both: author in a grid (or a TMS such as Crowdin/Lokalise), then
**compile** to per-language runtime tables (Unity String Tables per locale, Unreal `.locres`,
gettext `.mo`). For LoveTactics — hand-edited Lua, ~2 languages, no TMS — we maintain a single
artifact, and the grid is the better single artifact.

### Trade-offs

| Concern | Columns grid (proposed) | Per-language files (current) |
|---|---|---|
| English context for translators | inline, same row | via generated `-- EN:` comment |
| Coverage / missing at a glance | blank cell in the row | needs the diff test (we have it) |
| Concurrent translators | all edit one file → merge conflicts | isolated per file |
| Runtime memory | loads all languages | loads only the active one |
| Adding a language | add a column to every row | copy `en.lua` → `fr.lua` |
| Whole-game localization | one catalog (split by namespace as it grows) | N files per namespace |

At this project's scale the two "cons" of the grid are weak: a few thousand short strings across a
handful of languages is single-digit MB (irrelevant on desktop LÖVE — studios split per-language at
runtime for language DLC/patching and TMS export, not memory), and merge conflicts only bite with
*concurrent* translators, which a small team rarely has.

**Verdict:** adopt the columns grid as the hand-edited catalog. It matches the mental model, removes
the `-- EN:` comment crutch, and makes gaps visible, at negligible cost now.

## Non-negotiables (keep these true)

1. **English is authored inline** in the conversations and remains authoritative. The grid's `en`
   column is a *generated mirror* for translator context — never the place English is edited.
   (Editing English only in the grid would drag us back to the unreadable `line="1"` authoring.)
2. **Split by namespace as it grows.** Start with one `data/lang/strings.lua`; when it gets large,
   split into `data/lang/strings/conversations.lua`, `.../items.lua`, `.../ui.lua`, each a grid,
   merged at load. Avoid one file of thousands of rows.

## Target schema

- **File:** `data/lang/strings.lua` returns a keyed map `{ [id] = { en = ..., ja = ..., ... } }`.
  Keyed map (not a header-row array) so lookups stay O(1).
- **Columns:** one per language code; `en` is always present (the mirror). A missing/`""` cell means
  "untranslated" → runtime falls back to `en`.
- **Ids:** unchanged (`Locale.key.*`). Rows sorted by id for stable diffs.

## Runtime

`models/locale.lua`:
- Load `data/lang/strings.lua` once (memoized). `Locale.languages()` derives the language list from
  the union of column names present (plus `en`).
- `Locale.get(key, fallback)` becomes: `en` → return `fallback` (inline is authoritative);
  otherwise `row = strings[key]; return (row and row[current]) or fallback`.
- `Locale.key` builders unchanged.

Callers are **unchanged** — `ui/dialogue.lua` and `models/conversation.lua` already go through
`Locale.get`. This is a format swap behind a stable seam.

### Runtime tokens

An authored line may carry these. Both are substituted by `Locale.substitute` **after** `Locale.get`
returns, so a token travels through the translated string and a translator is free to move it
wherever their grammar needs it. Keep the braces and the spelling exactly; a dropped token silently
prints nothing useful.

| Token | Becomes | Notes |
|---|---|---|
| `{name}` | The name the player typed at character creation | Falls back to the avatar blueprint's "Stranger" |
| `{select}` | The button to press. Keyboard/gamepad resolve to a drawn key cap (`Enter`, `A`); the mouse has no key to draw, so it becomes the plain word `Click` | Re-resolves every draw |

`{select}` exists because this project supports **mouse, keyboard and gamepad equally**, so an
instruction that says "Click" is wrong on two of them. Any line that tells the player to *do*
something — the tutorial's coaching bubbles are the current callers — must use it rather than naming
a device. `tests/tutorial_spec.lua` fails the build over a coaching line that hard-codes "click".

It is written as a **sentence opener**, reading straight into a preposition, as in
`"{select} on the imp to move in range."` So the same line resolves two ways:

```
keyboard / gamepad   [Enter]  on the imp to move in range.     ← key drawn as a cap
mouse                Click on the imp to move in range.        ← words, no cap
```

The mouse is the deliberate exception. "Click" is **not a key** — drawn as a cap it invents a button
nobody owns, and a player who goes looking for it on their mouse has been handed a puzzle by the one
thing on screen whose job was to be unambiguous. `Locale.selectKey()` returns `nil` there, and a nil
is what tells a caller to render words instead of a pill (`Locale.coachLine` → `ui/coach_bubble.lua`).

A line that needs the token mid-sentence will get a stray capital — reword it, or add a lowercase
companion token the day a language genuinely needs one.

## Extraction tool

`tools/extract_strings.lua` (`extract-strings`):
- Still stamps stable `tag`s into conversation files and collects `{ key, en }` records.
- Instead of writing `en.lua` + per-language files, **merge one `strings.lua`**: for each record,
  set/overwrite the `en` cell from the inline English (the mirror), and **preserve** every other
  language cell already present for that id. New ids get their non-`en` cells blank. Emit rows
  sorted by id, each on a line, e.g. `["line.wrath_intro.1"] = { en = "...", ja = "..." },`.
- **One-time migration:** read the existing `data/lang/en.lua` / `ja.lua`, fold them into the grid
  by id (en → `en` cell, ja → `ja` cell), then delete the old files. The tool can do this in a
  single run since the ids already match.
- Keep the safety net: validate generated Lua with `loadstring` before writing; write into the
  project source tree via `io.open` on `love.filesystem.getSource()` (LÖVE's save dir is read-only
  to us here).

### Staleness (unchanged limitation)

Neither model auto-detects a translation gone stale because its English changed (that needs a stored
per-cell source snapshot). The grid makes it *visible*: `en` and `ja` sit in the same row, so a
mismatch is easy to spot — strictly better than the current `-- EN:` comment, at no extra cost. A
future enhancement could store a hidden `en@` hash per translated cell and have `extract-strings`
flag rows whose `en` moved; out of scope here.

## When to revisit

If a **TMS or a shared translation spreadsheet** becomes the source of truth, or **multiple
concurrent translators** come on, move the grid *out* of the repo into that tool and have
`extract-strings` **compile it down to per-language runtime files** (the compiled form, generated —
not hand-edited). Choosing the grid now does not burn that bridge: the extraction tool is the seam
that can emit either shape.

## Migration (done)

The switch from per-language files to the grid was executed:

1. `extract-strings` builds/merges `data/lang/strings.lua`, folding in the old `en.lua`/`ja.lua` and
   deleting them (`os.remove`, a no-op on later runs).
2. `models/locale.lua` loads the grid; `get`/`languages`/`raw` read columns; key builders unchanged.
3. `extract-strings` ran and round-tripped (ja values preserved, `en` mirrors the inline source).
4. `tests/locale_spec.lua` checks each row has an `en` cell, blank cells fall back, and columns are
   reported; `tests/conversation_spec.lua` verifies the `en` cell is in sync with the inline line.
5. Verified in-game (English + Japanese) with a CJK font at `assets/fonts/ui.ttf` (see
   `assets/fonts/README.md`).

## Verification

- Headless: `& "E:\LOVE\lovec.exe" . test` — `locale_spec` + `conversation_spec` green, proving the
  grid loads, columns resolve, and the `en` mirror is in sync with inline English.
- In-game: start the Colosseum's debut quest; the `wrath_intro` scene renders in English, and with
  `Locale.set("ja")` renders Japanese (title, name plate, lines, branching choices) — identical to
  today, since only the catalog's on-disk shape changed.
